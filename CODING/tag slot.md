# 🎯 Разница между <slot /> и props
В Svelte (и в Svelte 5) есть четкое разделение между тем, как передается данные (props) и как передается содержимое (slot).
1. props (Свойства)
    • Назначение: Передача данных (строк, чисел, объектов, функций) из родителя в дочерний компонент.
    • Синтаксис: Передаются как атрибуты в теге компонента и принимаются в <script> с помощью $props() (Svelte 5) или export let (Svelte 4).
    • Пример:
      HTML
      <Button handler={startGame} label="Начать Игру 🚀" />
      Здесь label — это проп, содержащий строку.
2. <slot /> (Слот)
    • Назначение: Передача HTML-контента (разметки, других компонентов, текста) из родителя в определенное место в дочернем компоненте.
    • Синтаксис: Контент помещается между открывающим и закрывающим тегами дочернего компонента.

🖼️ Как работает <slot />
Когда вы пишете:
Svelte
<Button handleClick={startGame}>
  Начать Игру 🚀
</Button>
Svelte берет все, что находится внутри тегов <Button>...</Button> (то есть Начать Игру 🚀), и вставляет этот контент туда, где в компоненте Button.svelte находится тег <slot />.
A. Просто Текст
Если вы передаете просто текст:
HTML
<Button>Начать Игру 🚀</Button>
То в скомпилированном HTML это будет выглядеть так:
HTML
<button class="styled-button">
  Начать Игру 🚀  </button>
B. Произвольная HTML-Разметка
Если вам нужна разметка (например, <span>START</span>), вы также помещаете её внутрь тегов, и она будет передана точно так же:
HTML
<Button>
  <span>START</span>  </Button>
Результат в HTML:
HTML
<button class="styled-button">
  <span>START</span>  </button>
Таким образом, <slot /> — это мощный механизм для композиции, позволяющий вашему компоненту Button.svelte быть универсальной оберткой, которая может содержать любой контент, от простого текста до сложной вложенной структуры.

#   Механика вложения контента (<slot />) для +layout.svelte


Механизм, который вы описываете, называется вложением макетов (Layout Nesting), и он является фундаментальной частью работы файловой маршрутизации в SvelteKit.

Вот пошаговая механика, как контент страницы (home/account/+page.svelte) попадает в макеты:

1. Начальная точка: Страница
SvelteKit знает, что для URL /home/account должна быть отображена страница, определенная в файле src/routes/(home)/home/account/+page.svelte.

2. Подъем по иерархии (Слой 1: Макет группы)
SvelteKit начинает смотреть вверх по файловой структуре, ища родительские макеты (+layout.svelte):

Страница .../account/+page.svelte ищет свой ближайший родительский макет.

Он находит src/routes/(home)/+layout.svelte.

Весь контент из account/+page.svelte автоматически заменяет тег <slot /> в этом макете.

3. Подъем по иерархии (Слой 2: Глобальный Макет)
После того как макет (home)/+layout.svelte собрал свое содержимое, SvelteKit продолжает подъем:

Макет (home)/+layout.svelte ищет своего родителя.

Он находит src/routes/+layout.svelte (глобальный макет).

Теперь результат работы макета (home)/+layout.svelte (вместе с его хедером и футером) автоматически заменяет тег <slot /> в глобальном макете.

4. Финальный Рендеринг
Только после того как глобальный макет (который содержит в себе макет группы, который содержит в себе страницу) полностью собран, SvelteKit отправляет финальный HTML в браузер.
