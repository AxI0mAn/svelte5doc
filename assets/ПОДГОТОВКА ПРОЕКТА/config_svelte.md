# ⚙️ svelte.config.js: Конфигурация Компилятора Svelte
Этот файл отвечает за то, как Svelte-компилятор обрабатывает ваши компоненты (.svelte файлы) и как SvelteKit организует маршрутизацию и сборку.

Базовый минимум функциональности:

-------------------------------------------------
import adapter from '@sveltejs/adapter-static'; - Назначение: Правильный адаптер для вашего целевого хостинга (GitHub Pages).

-------------------------------------------------
fallback: '200.html' - Это критически важная настройка для SPA. Она гарантирует, что если пользователь заходит по несуществующему пути, сервер возвращает файл 200.html (или index.html), который позволяет SvelteKit на стороне клиента корректно обработать маршрутизацию, избегая ошибки 404.

-------------------------------------------------
compilerOptions: { runes: true, } - 	Включение/выключение Runes Mode (новый реактивный синтаксис Svelte 5).
	Критически важно! Если вы включаете runes: true здесь, он применяется ко всему проекту.

-------------------------------------------------
kit.prerender	- Настройка статического рендеринга (SSR) страниц. Вы можете указать, какие страницы нужно сгенерировать во время сборки.	
    Важно для SEO и производительности. Если ваш сайт в основном статический (как блог или викторина), вы будете это использовать.
    Если у вас много статического контента, вы можете явно настроить, какие страницы должны быть пререндерены для лучшей производительности, хотя для SPA с fallback это не всегда строго необходимо.
 
-------------------------------------------------
kit.files	- Указывает SvelteKit, где искать ключевые папки (например, assets, routes, lib).	
    Полезно, если вы хотите изменить стандартную структуру папок.

-------------------------------------------------
kit.alias	- Создание алиасов (псевдонимов) для импорта (например, $lib). SvelteKit использует это для автоматической настройки, но вы можете добавить свои.	
    Удобно для чистого кода: вместо ../../../components вы пишете $components/MyComponent.

------------------------------------------------
kit.paths.base: Если ваш сайт размещен не в корне домена, а в подпапке (например, github.io/my-repo), вам нужно будет настроить эту опцию.

Настройка kit.paths.base актуальна и критически важна для проектов SvelteKit, развернутых на GitHub Pages, если вы используете репозиторий проекта не в корневом каталоге своего домена. 

Что делает paths.base:
1. SvelteKit автоматически преобразует все ваши ссылки <a href="/home"> в <a href="/my-svelte-app/home">.
2. Пути к статическим ресурсам (<img src="/logo.png">) будут преобразованы в /my-svelte-app/logo.png.
3. Без этой настройки браузер будет пытаться найти ресурсы и страницы в корне домена (https://username.github.io/home), что приведет к ошибкам 404, так как фактический контент находится в подпапке /my-svelte-app/.

# Сайт проекта (Репозиторий) по адресу username.github.io/my-repo-name - Сайт находится в подкаталоге (/my-repo-name)!!!

Поскольку большинство проектов SvelteKit, размещенных на GitHub Pages, используют схему Сайт проекта, вам нужно сообщить SvelteKit, что он должен добавлять префикс ко всем внутренним ссылкам и ресурсам.

Поэтому нужно добавить код const base_path = process.env.VITE_BASE_PATH || ''; сразу при разработке. Это позволяет вам безопасно разрабатывать локально с пустым базовым путем и корректно собирать проект для GitHub Pages с использованием переменной окружения.

# 🛠️ Пример настройки
определяете base_path в svelte.config.js, но делаете его зависимым от переменной окружения (VITE_BASE_PATH или NODE_ENV).

Если ваш репозиторий называется my-svelte-app, и сайт доступен по адресу https://username.github.io/my-svelte-app/, ваша конфигурация должна выглядеть так:


// svelte.config.js

// ⭐️ Читаем переменную окружения
// Если переменная окружения установлена, используем её. 
// Иначе (в режиме разработки), используем пустую строку, что соответствует корню.
const base_path = process.env.VITE_BASE_PATH || '';

# 🚀 Этапы запуска и сборки

A. Локальная разработка (npm run dev)

Команда: npm run dev

Переменная: process.env.VITE_BASE_PATH не установлена.

Результат: base_path становится '' (пустая строка).

Пути: Ссылки генерируются как /home, /game1. Все работает на localhost:5173.



B. Сборка для GitHub Pages (npm run build)

Команда: Вы должны запустить сборку, установив переменную окружения.

В Linux/macOS:

Bash

VITE_BASE_PATH='/my-repo-name' npm run build

# 🤝 Взаимодействие файлов
Важно понимать, что SvelteKit работает поверх Vite:

Когда вы запускаете проект, Vite запускается первым, используя свою конфигурацию (vite.config.js).

Vite подключает плагин SvelteKit, который, в свою очередь, считывает настройки из svelte.config.js.

Таким образом, vite.config.js управляет общим процессом сборки и инструментами, а svelte.config.js — тем, как Svelte обрабатывает специфические .svelte файлы.


=====================================
# 1. 🌐 Конфигурация SvelteKit (svelte.config.js)

Этот файл отвечает за маршрутизацию, рендеринг и адаптацию проекта к различным хостингам.

A. Примеры kit.prerender и разных адаптеров

JavaScript

// svelte.config.js

import adapterStatic from '@sveltejs/adapter-static'; // Для статического хостинга (Netlify, Vercel Static, GitHub Pages)
import adapterNode from '@sveltejs/adapter-node';   // Для Node.js серверов (DigitalOcean, Heroku)

// Выбираем адаптер, который будем использовать:
const activeAdapter = process.env.NODE_ENV === 'production' ? adapterStatic : adapterNode;

/** @type {import('@sveltejs/kit').Config} */

const config = {
  // ... настройки препроцессора

  kit: {
    // 1. Настройка Адаптера
    adapter: activeAdapter({
      // Специфические настройки для выбранного адаптера (например, куда выводить статику)
      pages: 'build',
      assets: 'build'
    }),

    // 2. Настройка Pre-rendering (применяется, если используется adapter-static)
    prerender: {
      // Глубина обхода: сколько уровней ссылок нужно обойти, начиная с корня
      // Если у вас небольшой сайт, можно установить true (рекомендуется)
      default: false, 
      
      // Исключить страницы из пререндеринга (например, динамические роуты)
      // Роут /admin/.* не будет статически сгенерирован
      // (Например, можно использовать: ['/admin', '/user/*'])
      entries: ['*', '!/admin', '!/dashboard/*'], 
      
      // Игнорировать страницы с ошибками (4xx) при сборке
      handleHttpError: 'fail', 
    },

    // ... другие настройки
  }
};

export default config;

B. Настройка kit.alias (SvelteKit)
Для создания алиасов, которые работают как в SvelteKit (для маршрутизации, хуков), так и в Vite (для импортов).

JavaScript

// svelte.config.js (продолжение)
// ...

const config = {
  // ... настройки препроцессора

  kit: {
    // 1. Адаптер и Pre-rendering (как показано выше)

    // 2. Настройка Алиасов
    alias: {
      // Создаем алиас $components для src/components
      // Теперь вы можете импортировать: import MyComponent from '$components/MyComponent.svelte';
      '$components': 'src/components',
      
      // $stores уже настроен по умолчанию для $lib/stores
      '$stores': 'src/lib/stores' 
    }
  },

  // ...
};

export default config;

C. Текущая версия - базовый минимум.

import adapter from '@sveltejs/adapter-static'; // Импортируем адаптер для статического SPA размещенного на gitHub

import { vitePreprocess } from '@sveltejs/vite-plugin-svelte'; // <-- Импортируем для работы с SCSS

/** @type {import('@sveltejs/kit').Config} */
const config = {
	// Добавляем секцию preprocess и используем vitePreprocess()
	preprocess: vitePreprocess(), // <-- Используем

	kit: {
		adapter: adapter({
			// 1. Настройка для SPA fallback
			fallback: '200.html',

			// 2. Настройка папок вывода (по умолчанию 'build', но лучше явно указать)
			// Куда поместить сгенерированные HTML-страницы
			pages: 'build',
			// Куда поместить статические ресурсы (CSS, JS, картинки)
			assets: 'build'
		})
	},

	// Активация Runes Mode глобально:
	compilerOptions: {
		runes: true,
	}
};

export default config;
