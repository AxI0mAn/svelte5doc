Шпаргалка: Полный цикл разработки (Feature Workflow)

Этот алгоритм описывает путь от идеи новой функции до 
её появления на живом сайте с использованием GitHub Actions (CI/CD).

Этап 1: Подготовка и создание ветки

Никогда не делайте правки сразу в main, если проект уже задеплоен.

    1. Обновите локальный main:
        git checkout main
        git pull origin main

    2. Создайте ветку для новой задачи (например, feat-pwa):
        git checkout -b feat-pwa

Этап 2: Разработка и локальная проверка

    1. Пишите код, добавляйте файлы (static/sw.js, manifest и т.д.).

    2. Проверьте сборку локально:
        npm run build
        npm run preview

Если в preview всё работает (PWA устанавливается, пути верные) — можно отправлять.

Этап 3: Пуш и Pull Request (PR)

    1. Добавьте изменения в индекс:
        git add .

    2. Сделайте коммит:
        git commit -m "feat: добавлена поддержка PWA и сервисного рабочего"

    3. Отправьте ветку на GitHub:
        git push origin feat-pwa

    4. Создайте Pull Request: Зайдите в репозиторий на GitHub, нажмите "Compare & pull request". 
    Проверьте изменения и нажмите "Create pull request".

Этап 4: CI/CD и Слияние (Merge)
    
    1. GitHub Actions: После создания PR или пуша в main, во вкладке Actions запустится ваш воркфлоу (deploy.yml).
        ◦ Дождитесь зеленой галочки.
    
    2. Слияние: Если тесты/сборка прошли успешно, нажмите "Merge pull request" на странице PR.
    
    3. Удаление ветки на GitHub: После мерджа GitHub предложит кнопку "Delete branch" — нажмите её.

Этап 5: Финальная очистка локально

Теперь нужно привести локальный компьютер в соответствие с тем, что на сервере.

    1. Вернитесь в main: 
        git checkout main

    2. Заберите изменения из облака: 
        git pull origin main

    3. Удалите локальную временную ветку: 
        git branch -d feat-pwa

(Если Git ругается, что ветка не объединена, но вы уверены в успехе мерджа — используйте -D).

    4. Синхронизируйте список всех веток (удалите "призраки" удаленных веток):
        git fetch --prune


Полезные команды "на всякий случай"

    • Отмена последнего коммита (с сохранением кода): 
            git reset --soft HEAD~1
    • Просмотр статуса: 
            git status
    • Просмотр веток: 
            git branch -a
    • Если забыли .nojekyll: Убедитесь, что в static/ лежит пустой файл .nojekyll, иначе GitHub Pages сломает пути SvelteKit.
    • Зачем нужен .nojekyll?: GitHub Pages по умолчанию использует движок Jekyll, который игнорирует любые папки, начинающиеся с нижнего подчеркивания (например, _app в SvelteKit). Файл .nojekyll отключает этот движок. Если ваш скрипт деплоя в GitHub Actions (например, JamesIves/github-pages-deploy-action) имеет параметр dotfiles: true, он может создавать этот файл автоматически. Однако лучше иметь его в static/, чтобы гарантировать работоспособность PWA и загрузку стилей.
