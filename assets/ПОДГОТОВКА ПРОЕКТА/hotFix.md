Есть основная ветка main и создана новая triton для реализации нового функционала, но после последнего релиза нашлись ошибки, которые необходимо исправить. Как правильноэто сделать?

Правильный подход в профессиональной разработке называется Hotfix (срочное исправление). Вот самый чистый алгоритм действий:

Рекомендуемый алгоритм: ветка fix
Переключитесь на main и обновите её:
git checkout main -> git pull origin main.

Создайте ветку для исправлений:
git checkout -b fix-bugs. Исправьте ошибки, сделайте коммит и пуш.

Слейте исправления в main:
Через Pull Request на GitHub или командой git merge fix-bugs внутри main. Теперь ваш сайт на GitHub исправлен и работает корректно.

Как «подтянуть» исправления в ветку triton?
Вам не нужно удалять triton. Вам нужно просто влить в неё обновленную main.

git checkout triton

git merge main

Git сам добавит исправленные файлы в вашу новую ветку, не затронув тот функционал, который вы там уже начали писать.


Краткая схема ваших действий (Cheat Sheet):

Шаг 1
git checkout main — Уходим с ветки triton на главную.

Шаг 2
git pull origin main — Важно! Забираем актуальное состояние main из облака, чтобы исправления базировались на последней версии сайта.

Шаг 3
git checkout -b toFixBugs — Создаем ветку для ремонта.

Шаг 4
...исправляем код и ТЕСТИРУЕМ его локально (через npm run dev или npm run preview)...

Шаг 5
git commit -am "fix: description of fix" — Сохраняем исправления в локальный коммит.

Шаг 6
git push origin toFixBugs — Отправляем ветку с ремонтом на GitHub.

Шаг 7 (в браузере на GitHub)
Нажимаем кнопку "Compare & pull request", затем "Create pull request". Проверяем, нет ли конфликтов.

Шаг 8 (в браузере на GitHub)
Нажимаем "Merge pull request" и затем "Confirm merge". Теперь ваши исправления официально в ветке main и сайт обновляется.

Шаг 9 (в браузере на GitHub)
Нажимаем "Delete branch" (фиолетовая кнопка). В облаке ветка toFixBugs больше не нужна.

Шаг 10 (возвращаемся в терминал)
git checkout main — Снова переходим в main.

Шаг 11
git pull origin main — Забираем в локальный main то самое слияние, которое мы только что сделали на GitHub.

Шаг 12
git branch -d toFixBugs — Удаляем ветку с ремонтом локально (она уже влита в main).

Шаг 13
git checkout triton — Возвращаемся к работе над новым функционалом.

Шаг 14
git merge main — Критический шаг! Вливаем исправленный main в triton. Теперь в вашей новой ветке есть и ваши наработки по функционалу, и свежие исправления ошибок.
После слияния main в triton в терминале может появиться окно для ввода сообщения коммита — 
просто введите :q и нажмите Enter, если работаете в Vim, или просто закройте вкладку редактора, если он открылся.

Итог:
Теперь у вас идеальный порядок. Ветка toFixBugs удалена и не мозолит глаза, main содержит чистый исправленный код, а triton — самый актуальный фундамент для продолжения работы. Если при git merge main (Шаг 14) возникнут конфликты (если вы правили одни и те же строки), Git предложит их решить — это нормально.
